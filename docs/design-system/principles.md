# Principles — Voice & 사용 원칙

Toss 디자인 언어의 운용 철학을 7개 원칙으로 정리했습니다. 신규 화면을 만들기 전과 후에 셀프 리뷰 체크리스트로 사용하세요.

---

## 1. 가독성 우선

정보 위계는 **타이포그래피의 사이즈 차**로 표현합니다. 색이나 굵기는 보조 도구입니다.

- 안티패턴: 같은 사이즈에서 굵기·색만 바꿔 헤더와 본문을 구별하려는 경우.

## 2. 하나의 위계, 하나의 색

한 화면에서 `blue500`(Primary) 사용은 **핵심 액션 1곳**으로 제한합니다. 강조가 두 곳이면 둘 다 약해집니다.

- 안티패턴: "확인", "취소", "더보기"가 모두 파란색인 경우.

## 3. 여백이 위계다

같은 그룹은 **8 이하**로 묶고, 그룹 사이는 **24~48**로 넓힙니다. 여백 차이가 그룹의 경계입니다.

- 안티패턴: 모든 항목 간격이 16으로 동일해 그룹이 보이지 않는 경우.

## 4. 시스템 폰트 스케일 존중

사용자가 시스템 글자 크기를 키웠다면, **반드시 따라가야** 합니다. 폰트 사이즈를 px로 잠그지 않습니다.

- 안티패턴: 디자인 깨짐을 우려해 폰트 크기를 고정해 접근성 옵션을 무력화하는 경우.

## 5. 명도 대비

본문 텍스트는 **`grey800`~`grey900`**, 비활성·보조는 **`grey500`** 이상에서 멈춥니다. WCAG 기준 본문 4.5:1, 큰 텍스트 3:1 대비를 확보합니다.

- 안티패턴: 캡션을 `grey400` 이하로 두어 읽기 어렵게 만드는 경우.

## 6. 상태 색은 의미로만

`red` / `green` / `yellow` / `orange`는 **상태 전달**(오류·성공·경고·강조)에만 사용합니다. 장식·카테고리 식별에는 액센트(teal/purple)나 grey를 씁니다.

- 안티패턴: 단순한 카테고리 칩에 `red500`을 사용해 사용자가 오류로 오인하는 경우.

## 7. 장식 금지

그림자·라운드·테두리는 **기능을 가질 때만** 사용합니다(떠 있음, 영역 구분, 인터랙션 가능 표시). 그 외에는 제거합니다.

- 안티패턴: 모든 카드에 짙은 그림자를 적용해 화면이 평탄해지는 경우.

## 8. 호스트와 구분

우리 서비스가 외부 호스트(웹뷰, 슈퍼앱 미니앱, 임베드) 위에 올라갈 때 호스트 UI와 시각적으로 충돌하지 않도록 합니다.

- 안티패턴: 호스트 하단 탭과 똑같은 형태·위치의 자체 탭바를 띄워 사용자가 자기 위치를 잃는 경우.

## 9. 위치 인지

사용자가 어떤 화면에 있는지 항상 명확해야 합니다. 헤더, 탭, 모달 헤더가 그 신호입니다.

- 안티패턴: 모달과 풀스크린 화면이 시각적으로 동일해 어디로 돌아가야 할지 모르는 경우.

## 10. IA 단순성

하단 탭은 **2~5개 사이**로 유지합니다. 그 이상은 더 깊은 IA(섹션·서브내비) 신호입니다.

- 안티패턴: 6개 이상 탭으로 정보가 평탄화되어 우선순위가 사라지는 경우.

---

## 셀프 리뷰 체크리스트

새 화면을 PR로 올리기 전 확인:

- [ ] Primary `blue500`을 사용한 액션이 화면당 1개 이하인가?
- [ ] 본문 텍스트가 `grey800`~`grey900` 범위인가?
- [ ] 그룹 간 여백이 항목 간 여백보다 명확히 큰가? (대략 3배 이상 권장)
- [ ] 모든 폰트 사이즈가 토큰을 통해 들어오는가? (px 직접 입력 없음)
- [ ] 상태 색이 장식 용도로 쓰이지 않았는가?
- [ ] 그림자가 "떠 있음"의 정보를 전달하는가, 아니면 단순 장식인가?
- [ ] 다크 모드 또는 글자 크기 확대 시 위계가 유지되는가?
- [ ] 호스트 위에 올려놓고 봤을 때 우리 영역이 시각적으로 구분되는가?
- [ ] 사용자가 현재 화면이 모달인지 풀스크린인지 한눈에 알 수 있는가?
- [ ] 하단 탭(있다면) 개수가 2~5개 범위 안에 있는가?
